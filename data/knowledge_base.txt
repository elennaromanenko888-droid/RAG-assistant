RAG расшифровывается как Retrieval-Augmented Generation. Это подход, при котором приложение сначала ищет релевантные фрагменты в базе знаний, а затем использует найденный контекст для ответа пользователю. Такой подход помогает отвечать на вопросы по собственным документам, даже если большая языковая модель не знает эти документы заранее.

В типичном RAG-пайплайне есть несколько шагов. Сначала мы загружаем исходный текст из файла или другого источника. Затем текст делится на небольшие части, которые называются чанками. После этого для каждого чанка строится эмбеддинг, то есть числовой вектор, который приблизительно описывает смысл текста. Эти векторы сохраняются в векторной базе данных.

Чанкинг нужен потому, что большие документы неудобно искать и передавать в модель целиком. Если разбить текст на чанки по 100-300 слов, система сможет найти именно тот фрагмент, который связан с вопросом. Небольшое перекрытие между чанками помогает не потерять смысл на границе двух соседних фрагментов.

Эмбеддинги превращают текст в набор чисел. Похожие по смыслу тексты получают похожие векторы, поэтому по вопросу пользователя можно найти ближайшие фрагменты документа. В промышленной системе для эмбеддингов часто используют модели вроде sentence-transformers или API внешнего провайдера. В учебном проекте можно использовать простой локальный вариант, чтобы понять общий принцип без API-ключей.

ChromaDB - это векторная база данных для хранения эмбеддингов и поиска похожих документов. В ChromaDB можно создать коллекцию, добавить в нее документы, метаданные и эмбеддинги, а затем выполнить запрос и получить наиболее похожие чанки. Для небольших проектов ChromaDB удобно запускать локально в папке проекта.

Когда пользователь задает вопрос, RAG-ассистент тоже превращает вопрос в эмбеддинг. После этого векторная база сравнивает эмбеддинг вопроса с эмбеддингами чанков и возвращает самые близкие фрагменты. На основе этих фрагментов ассистент может показать найденный контекст или сформировать ответ.

Качество RAG-системы зависит от качества исходных документов, размера чанков, способа построения эмбеддингов и количества найденных фрагментов. Если чанки слишком большие, в них много лишней информации. Если чанки слишком маленькие, они могут потерять контекст. Поэтому параметры chunk_size и overlap обычно подбирают экспериментально.
